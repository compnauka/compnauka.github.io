# AGENTS.md — інструкції для Codex / Claude Code / інших агентів

## 1. Контекст проєкту

Це дитяча інтерактивна книга “Привіт, ШІ!” українською мовою. Проєкт має бути простим, безпечним, доступним і придатним для школи.

Не перетворювати його на складний фреймворк-проєкт без потреби. Пріоритет — стабільна статична книга, яку легко розмістити на GitHub Pages / Cloudflare Pages / звичайному static hosting.

## 2. Головні принципи

1. Дитяча безпека важливіша за аналітику й маркетинг.
2. Accessibility важливіша за декоративні ефекти.
3. Контент має бути редагованим без знання HTML.
4. Код має бути простим і зрозумілим.
5. Не ламати поточний візуальний стиль без окремого завдання.
6. Кожна зміна має бути перевірена вручну в браузері.

## 3. Заборонено

- Додавати Google Analytics, Meta Pixel, рекламні скрипти або інші трекери.
- Блокувати zoom через `user-scalable=no` або `maximum-scale=1`.
- Просити дитину вводити персональні дані.
- Зберігати персональні дані в `localStorage`.
- Додавати довільний HTML у контент без sanitizer/рендерера.
- Підключати важкі фреймворки без прямої потреби.
- Переписувати весь дизайн, якщо завдання стосується лише архітектури або безпеки.
- Видаляти accessibility-атрибути.
- Прибирати focus outline без рівноцінної заміни.
- Залишати невалідні `.js` файли.

## 4. Обов’язковий перший пакет виправлень

Перед великим рефакторингом зробити hotfix:

1. У `index.html` замінити viewport на:

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```

2. Видалити блоки Google Analytics і Plausible.
3. Додати skip-link.
4. Додати `id="main-content"` до `<main>`.
5. Додати ARIA для меню:
   - `aria-controls="sidebar"`
   - `aria-expanded="false"`
6. Додати Escape-закриття меню.
7. Додати `prefers-reduced-motion` у CSS.
8. Усі зовнішні посилання з `target="_blank"` доповнити `rel="noopener noreferrer"`.
9. Вирішити долю `quizzes_updated.js`: інтегрувати або перейменувати.

## 5. Робота з `innerHTML`

Поточний код використовує `innerHTML`. Не збільшувати кількість таких місць.

Для нового коду:

- текст — тільки через `textContent`;
- елементи — через `document.createElement`;
- посилання — через DOM API;
- медіа — через DOM API;
- feedback — через DOM API.

Дозволений тимчасовий виняток: статичні SVG-іконки, якщо вони не містять даних із контенту.

## 6. Цільовий формат контенту

Поступово перейти від HTML-рядків до структурованого контенту.

Рекомендована форма повідомлення:

```js
{
  speaker: "tymko",
  blocks: [
    { type: "paragraph", text: "Розумку, я не розумію, що саме називати ШІ." }
  ]
}
```

Для Розумка:

```js
{
  speaker: "rozumko",
  blocks: [
    { type: "paragraph", text: "ШІ — це програма, яка може виконувати завдання, що зазвичай потребують людського мислення." },
    { type: "list", items: ["розпізнавати зображення", "пропонувати варіанти", "створювати тексти"] }
  ],
  image: {
    src: "005.webp",
    alt: "Тимко дивиться на приклади ШІ навколо себе."
  }
}
```

## 7. Сюжетна ціль

Оновлювати книгу під наскрізну логіку:

> Тимко готується до шкільного Тижня ШІ.

Кожен розділ має відповідати на питання: “Що саме Тимко підготував після цього розділу?”

Приклади:

- плакат;
- список перевірених фактів;
- демонстрація з картками;
- запрошення;
- правила безпеки;
- гра для першачків;
- фінальна доповідь.

## 8. Машинне навчання

Не повертати конкурс із математики як основну ситуацію. Нова рекомендована ситуація:

> Тимко готує демонстрацію для Тижня ШІ: як навчити комп’ютер відрізняти “кіт”, “пес” і “не тварина”.

Пояснити:

- приклади;
- ознаки;
- помилки;
- покращення через різноманітні дані.

## 9. Безпека в контенті

Не писати дітям, що “справжній ШІ ніколи не попросить дані”. Коректніше:

> Якщо сайт, бот або незнайомець просить фото, адресу, номер телефону чи пароль — не надсилай самостійно. Зупинись і покажи дорослому.

## 10. Порядок рефакторингу

Рекомендований порядок:

1. Hotfix privacy/accessibility.
2. Валідація синтаксису JS.
3. Винесення рендерерів із `main.js`.
4. Структурований формат контенту.
5. Переписування сюжетної сітки.
6. PWA/offline.
7. Фінальне тестування.

Не робити все в одному величезному коміті, якщо є можливість розбити.

## 11. Код-стиль

- Використовувати ES-модулі.
- Імена функцій — дієслівні: `renderQuiz`, `closeMenu`, `saveProgress`.
- Не використовувати глобальні змінні без потреби.
- Не створювати приховану магію.
- Коментарі писати там, де рішення неочевидне.
- Не оптимізувати передчасно.

## 12. Перевірка після кожної зміни

Виконати:

```bash
node --check main.js
node --check bookSections.js
```

Після рефакторингу з модулями перевірити всі JS-файли:

```bash
find . -name "*.js" -not -path "./node_modules/*" -print0 | xargs -0 -n1 node --check
```

Ручний тест:

- відкрити обкладинку;
- перейти вперед/назад;
- відкрити меню;
- пройти квіз;
- перевірити мобільну ширину;
- перевірити консоль.

## 13. Критерії завершення задачі

Задача вважається завершеною, якщо:

- код запускається;
- немає помилок у консолі;
- немає синтаксичних помилок;
- виконано конкретні acceptance criteria;
- не додано нових privacy-ризиків;
- не зламано клавіатурну навігацію;
- контент не став складнішим для дітей.

## 14. Неочевидні рішення, які треба зберегти

- Таймери повідомлень треба очищати при переході між розділами.
- Квізи не мають карати дитину за помилку.
- Тон повідомлень має залишатися дружнім.
- Статичність проєкту — перевага, не недолік.
- Сайт має залишатися придатним для простого хостингу.
