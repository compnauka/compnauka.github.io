# ARCHITECTURE.md — архітектура інтерактивної книги “Привіт, ШІ!”

## 1. Поточна архітектура

Проєкт є статичним браузерним застосунком без бекенду.

Поточний потік роботи:

```text
index.html
  ├─ підключає Tailwind CDN, Google Fonts, style.css
  ├─ містить каркас: sidebar, chatWindow, bottomNav
  └─ підключає main.js як ES-модуль

main.js
  ├─ імпортує bookSections з bookSections.js
  ├─ формує список розділів
  ├─ рендерить навігацію
  ├─ рендерить поточний розділ у chatWindow
  ├─ додає повідомлення з затримкою
  ├─ обробляє квізи
  ├─ відкриває/закриває мобільне меню
  └─ зберігає прогрес у localStorage

bookSections.js
  ├─ містить обкладинку
  ├─ містить вступ і розділи
  ├─ містить HTML-фрагменти в полях content
  ├─ містить посилання на images/audio
  └─ містить квізи
```

## 2. Що в поточній архітектурі добре

- Немає серверної частини — менше ризиків для дитячого продукту.
- Уся логіка зрозуміла й зібрана в кількох файлах.
- Використано ES-модуль `main.js`.
- Контент формально відокремлений у `bookSections.js`.
- Для невеликого прототипу структура проста й підтримувана.

## 3. Архітектурні проблеми

### 3.1. Контент змішаний із HTML

У `bookSections.js` текст часто зберігається як HTML:

```js
content: `<p class="mb-2">...</p><ul>...</ul>`
```

Це ускладнює:

- редагування текстів;
- перевірку орфографії;
- локалізацію;
- автоматичну валідацію контенту;
- безпечний рендеринг;
- роботу агентів, які можуть випадково зламати HTML.

### 3.2. Багато `innerHTML`

`main.js` активно використовує `innerHTML` для навігації, повідомлень, квізів, feedback-блоків і медіа. Зараз джерело даних локальне, тому ризик контрольований, але для майбутнього це слабке місце.

### 3.3. Один файл відповідає за забагато речей

`main.js` одночасно відповідає за стан застосунку, навігацію, рендеринг, меню, квізи, сховище, таймери й медіа. Це нормально для прототипу, але далі файл буде складно підтримувати.

### 3.4. Немає схеми контенту

Зараз немає формального опису, яким має бути розділ, повідомлення, квіз, медіа або активність. Через це легко додати неповний або неправильний розділ.

## 6. Типи контентних блоків

Рекомендовані безпечні типи:

```js
{ type: "paragraph", text: "..." }
{ type: "list", items: ["...", "..."] }
{ type: "quote", text: "..." }
{ type: "note", title: "Запам’ятай", text: "..." }
{ type: "link", text: "...", href: "https://..." }
{ type: "image", src: "...", alt: "..." }
{ type: "audio", src: "...", title: "...", transcript: "..." }
```

Довільний HTML у контенті не використовувати. Якщо потрібен особливий блок, спочатку додати новий тип блоку в рендерер.

## 7. Стан застосунку

Поточний стан:

```js
let currentSectionIndex = 0;
```

Рекомендований стан:

```js
const appState = {
  currentSectionIndex: 0,
  activeQuizId: null,
  menuOpen: false,
  motionReduced: false
};
```

## 8. Сховище прогресу

Поточна реалізація використовує `localStorage`. Для шкільних комп’ютерів це може бути небажано, бо наступний учень бачить прогрес попереднього.

Рекомендовані варіанти:

- `sessionStorage`, якщо після закриття вкладки все має починатися спочатку;
- явний вибір “Почати спочатку / Продовжити з розділу N”.

## 9. Рендеринг

Для тексту використовувати `textContent`, а не `innerHTML`.

Добре:

```js
const p = document.createElement("p");
p.textContent = block.text;
```

Небажано:

```js
contentDiv.innerHTML = block.text;
```

`innerHTML` можна залишити тільки для статичних SVG-іконок, які не залежать від контенту. Усі користувацькі або контентні дані мають рендеритися через DOM API.

## 10. Таймери повідомлень

Поточна реалізація має `messageTimers` і правильно очищає таймери при переході між розділами. Це варто зберегти.

Покращення:

- не затримувати появу повідомлень, якщо користувач має `prefers-reduced-motion: reduce`;
- додати кнопку “Показати весь розділ одразу”;
- не робити занадто довгу каскадну анімацію для великих розділів.

## 11. Навігація

Поточний порядок розділів береться з `Object.keys(bookSections)`. Це працює, але краще перейти на масив:

```js
export const bookSections = [ ... ];
```

Масив явно зберігає порядок і простіший для валідації.

## 12. Квізи

Поточна реалізація показує випадковий квіз із розділу. Це добре для повторного проходження, але для педагогічної книги бажано:

- мати 1 основний контрольний квіз на розділ;
- додаткові питання показувати при повторному проходженні;
- не карати за помилки;
- пояснення показувати дружньо й коротко.

## 13. Пріоритет архітектурних змін

1. Прибрати довільний HTML із нового контенту.
2. Переписати рендер звичайних текстових повідомлень на DOM API.
3. Переписати квізи на DOM API.
4. Розділити `main.js` на модулі.
5. Додати валідацію структури контенту.
6. Додати PWA.

## 14. Критерії приймання рефакторингу

Рефакторинг вважається успішним, якщо:

- візуально книга залишилась такою ж або стала кращою;
- усі розділи відкриваються;
- квізи працюють;
- мобільне меню працює;
- у консолі немає помилок;
- `node --check` проходить для всіх JS-файлів;
- новий контент додається без написання HTML;
- зовнішній контент не може виконати JS;
- структура розділів перевіряється валідатором.
