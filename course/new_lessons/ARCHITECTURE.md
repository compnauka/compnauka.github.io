# ARCHITECTURE.md — архітектура інтерактивного підручника

Документ описує поточну архітектуру, ризики та цільовий напрям розвитку проєкту.

---

## 1. Архітектурний принцип

Проєкт має залишатися простим статичним підручником, який легко розгорнути на GitHub Pages або Cloudflare Pages і використовувати в школі без реєстрації.

Не потрібно переходити на складний фреймворк лише заради фреймворку. Для цього продукту важливіші:

- контрольована приватність;
- швидке завантаження;
- offline-first поведінка;
- зрозуміла структура уроків;
- мінімум залежностей;
- легкість аудиту.

---

## 2. Поточна модель

```text
HTML shell
  ↓
app.js + lesson-data.js
  ↓
lesson templates from js/lessons/*.js
  ↓
activity-registry.js
  ↓
task-*.js renderers
  ↓
shared state in memory
```

Головна сторінка використовує каталог уроків і карту модулів. Сторінки уроків є згенерованими HTML-файлами, які підтягують потрібний урок за `data-lesson-id`.

---

## 3. Сильні сторони

- Статичний сайт легко хостити.
- Немає backend-ризиків.
- Немає акаунтів і бази дитячих даних.
- Уроки відокремлені від рендера.
- Активності винесені в окремі модулі.
- Є PWA-основа.
- Є student/teacher mode.

---

## 4. Слабкі місця

### 4.1. Контентний контракт недостатньо суворий

Lesson data фактично перевіряється вручну. Потрібна схема і валідатор.

Перевіряти треба:

- required поля;
- унікальність `id`;
- коректність `activity.type`;
- відповідність `answer` доступним options;
- існування asset paths;
- коректність embedded-tool URL;
- текстові ліміти для student mode;
- coverage codes у каталозі ІФО;
- відсутність raw HTML у текстових полях.

### 4.2. Генерація сторінок і service worker розділені

Зараз HTML-сторінки генеруються скриптом, а `sw.js` оновлюється вручну. Це джерело помилок.

Ціль: один каталог уроків має генерувати:

- HTML сторінки;
- landing дані;
- service worker asset list;
- sitemap;
- coverage matrix.

### 4.3. Embedded tools потребують суворої політики

Інструменти мають бути локальними й same-origin. Зовнішні інструменти не дозволяються за замовчуванням.

---

## 5. Цільова модель

```text
content files
  ↓
lesson schema validation
  ↓
build/generation scripts
  ↓
static HTML + JS + CSS + PWA cache
  ↓
automated tests
  ↓
deploy
```

Архітектура має рухатися до “контент як дані”, але без зайвого backend.

---

## 6. Рекомендовані майбутні модулі

```text
scripts/validate-lessons.js або ps1
scripts/generate-service-worker-assets.js або ps1
scripts/generate-coverage-report.js або ps1
scripts/check-release-safety.js або ps1
```

Ці скрипти мають працювати без складної інфраструктури.

---

## 7. State management

Поточне правило:

- прогрес уроку — тільки в пам’яті сторінки;
- refresh скидає урок;
- довготривале збереження дитячих результатів заборонене.

Якщо в майбутньому з’явиться потреба в збереженні, це має бути окремий продукт або окремий teacher-controlled режим із політикою приватності, згодами, ролями та аудитом.

---

## 8. Інтерактивні активності

Кожна активність має бути:

- ізольованою;
- повторно використовуваною;
- доступною з клавіатури, де це можливо;
- придатною для touch;
- без залежності від зовнішніх сервісів;
- описаною в документації.

Не створювати новий тип активності без педагогічного обґрунтування.

---

## 9. PWA/offline

Поточна PWA-модель має бути посилена:

- всі сторінки уроків у precache;
- всі JS/CSS/tools/assets у precache;
- `CACHE_NAME` оновлюється при змінах;
- offline fallback не має приховувати проблему відсутнього ресурсу під час тестів;
- список assets бажано генерувати автоматично.

---

## 10. Архітектурні рішення, які не варто робити зараз

- Не додавати backend лише для прогресу.
- Не додавати логін учнів.
- Не переносити все в React/Vue/Svelte без чіткої користі.
- Не додавати важкі UI-бібліотеки.
- Не підключати зовнішні тренажери для дитячого режиму.
- Не ускладнювати teacher mode до CMS у межах цього ж статичного продукту.
