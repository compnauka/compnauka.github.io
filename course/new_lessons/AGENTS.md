# AGENTS.md — інструкція для Claude Code, Codex та інших ШІ-агентів

Цей файл є головною робочою інструкцією для агентів, які вносять зміни в інтерактивний підручник з інформатики для учнів 1–2 класів.

Проєкт орієнтований на дітей 6–8 років, шкільні комп’ютери, роботу без реєстрації та статичний хостинг. Будь-яка зміна має зберігати простоту, приватність, доступність і педагогічну доцільність.

---

## 1. Роль агента

Агент має працювати як обережний технічний редактор і розробник, а не як генератор випадкових покращень.

Перед кожною зміною агент зобов’язаний:

1. Зрозуміти навчальну мету зміни.
2. Перевірити, чи не порушує зміна безпеку та приватність дітей.
3. Перевірити, чи не ускладнює зміна інтерфейс для учнів 1–2 класів.
4. Зберегти поточну просту статичну архітектуру, якщо немає сильного обґрунтування для фундаментальної зміни.
5. Оновити документацію, тести та service worker у тому самому наборі змін, якщо зміна торкається уроків, інструментів, маршрутів, JS/CSS або PWA.

## 2. Непорушні правила

### 2.1. Приватність і спільні шкільні комп’ютери

- Не зберігати прогрес уроку, відповіді, спроби, дитячі результати або персональні дані між відкриттями сторінки.
- Після refresh або повторного відкриття браузера урок має стартувати з початку.
- `localStorage`, IndexedDB, cookies і серверне збереження заборонені для дитячого прогресу.
- `sessionStorage` можна використовувати лише для короткоживучих UI-налаштувань у межах поточної сесії, якщо це не розкриває роботу попереднього учня.

### 2.2. Заборонені файли в репозиторії та архівах

Ніколи не додавати в репозиторій, архів або pull request:

- `.browser-profile/`
- `tests/.browser-profile/`
- `Cookies`, `Safe Browsing Cookies`, `Login Data`, `History`, `Web Data`, `Local State`
- `.env`, приватні ключі, токени, службові профілі браузера
- кеші браузера, сесії, профілі Playwright/Chrome/Edge

Виявлена проблема: у попередньому архіві були браузерні профілі в корені та в `tests/`. Це вважається security/privacy blocker. Агент має видаляти такі файли та додавати перевірки, які не дозволяють їм повернутися.

### 2.3. Зовнішні залежності

- Не додавати CDN, трекери, зовнішні скрипти або сторонні віджети без окремого обґрунтування.
- Ядро підручника має працювати офлайн після кешування.
- Усі інструменти для дітей за замовчуванням мають бути локальними: `./tools/...`.

### 2.4. Безпека вбудованих інструментів

- `embedded-tool` має дозволяти лише same-origin URL з локальної папки `tools/`.
- Зовнішні URL заборонені за замовчуванням.
- Для `iframe` використовувати `sandbox`, `loading="lazy"`, `referrerpolicy="no-referrer"`.
- Для `window.open` використовувати `noopener,noreferrer`.
- `postMessage` приймати лише від довірених same-origin інструментів.

### 2.5. Контент для 1–2 класів

- Одна картка — одна навчальна дія.
- Одна інструкція — одна проста дія.
- Не перевантажувати урок текстом і кількістю вправ.
- Учнівський режим не повинен містити методичних пояснень для дорослих.
- Вчительські підказки мають бути приховані в teacher mode.
- Терміни пояснювати через життєві ситуації, а не через абстрактні визначення.

## 4. Рекомендована архітектурна лінія

Не переписувати проєкт на складний фреймворк без крайньої потреби. Поточний vanilla JS + статичні файли є практичним рішенням для шкільного підручника.

Цільове покращення має бути не ускладненням стеку, а переходом до суворого контентного контракту:

```text
lesson content → schema → validator → generator → tests → deploy
```

Пріоритетна архітектурна робота:

1. Описати JSON/JS-схему уроку.
2. Додати валідатор lesson data.
3. Генерувати HTML-сторінки та `sw.js` з єдиного каталогу.
4. Автоматично перевіряти assets, coverage, активності, URL і текстові ліміти.
5. Зберігати інтерактиви як незалежні повторно використовувані ядра.

## 5. Workflow перед зміною

Перед редагуванням агент має переглянути:

1. `AGENTS.md`
2. `SECURITY.md`
3. `PEDAGOGICAL_QUALITY.md`
4. `ARCHITECTURE.md`
5. релевантні файли в `js/`, `tools/`, `styles.css`, `lesson-page.template.html`

Потім агент має коротко визначити:

- які файли змінюються;
- чи торкається зміна контенту, архітектури, безпеки, PWA або UI;
- які тести треба оновити;
- чи треба перегенерувати HTML-сторінки;
- чи треба оновити `sw.js`.

---

## 6. Workflow для нового або зміненого уроку

1. Оновити або створити `js/lessons/<lesson-id>.js`.
2. Перевірити, що `id`, `title`, `studentHook`, `teacherOverview`, `coverage`, `objectives`, `sections`, `activities`, `quiz`, `reflection` відповідають шаблону.
3. Додати/оновити запис у `js/lessons/catalog.js`.
4. Оновити `js/landing-modules.js`, якщо змінюється модуль або порядок.
5. Оновити `scripts/generate-lesson-pages.ps1`.
6. Перегенерувати `m*-*.html`.
7. Оновити `sw.js`, щоб нова сторінка і всі потрібні JS/CSS/assets були в app shell cache.
8. Оновити документацію, якщо з’явився новий тип активності або нове правило.
9. Запустити тести.

---

## 7. Workflow для нового типу активності

Новий тип активності дозволено створювати лише тоді, коли наявні типи не підходять педагогічно.

Перед створенням нового `task-*.js` перевірити, чи задачу не можна реалізувати через:

- `classify`
- `transfer`
- `scenarios`
- `creative`
- `fill`
- `truefalse`
- `pick`
- `sequence`
- `draw`
- `key-trainer`
- `click-trainer`
- `trace-contour`
- `table-read`
- `embedded-tool`

Якщо новий тип справді потрібен:

1. Додати модуль `js/task-new-type.js`.
2. Зареєструвати його в `js/activity-registry.js`.
3. Додати документацію в `INTERACTIVE_ACTIVITY_ROADMAP.md` і `LESSON_TEMPLATE_GUIDE.md`.
4. Додати щонайменше один приклад у тестовому або реальному уроці.
5. Оновити `sw.js`.
6. Додати перевірки в `tests/run-tests.ps1`.
7. Перевірити клавіатуру, touch, screen reader basics, reduced motion.

## 8. Педагогічні ліміти для уроків

Для учнівського режиму:

- оптимально 4–6 основних карток на урок;
- 1 головна ідея на урок;
- 1–2 ключові поняття максимум;
- 1 основна інтерактивна дія для 1 класу;
- 2 інтерактивні дії максимум для 2 класу, якщо урок не розділений на частини;
- quiz короткий: 2–4 питання;
- інструкція до вправи: бажано до 12–16 слів;
- довге пояснення переносити в teacher mode або додатковий блок.

Якщо урок має 8+ активних блоків, агент має запропонувати поділ:

- обов’язкова частина;
- додатково;
- для швидких учнів;
- завдання з учителем.

## 9. UX-правила для дітей

- Primary action має бути помітною без прокручування, якщо це можливо.
- Кнопки мають бути великими, зрозумілими, з достатнім touch target.
- Не ховати важливі дії в незрозумілих меню.
- Не показувати дитині багато одночасних повідомлень.
- Успіх — теплий і короткий.
- Помилка — без звинувачення.
- Немає “сирих” технічних повідомлень.
- Canvas-завдання не мають бути єдиним способом пройти важливу навчальну дію.

---

## 10. Accessibility checklist

Кожна зміна інтерактиву має перевіряти:

- клавіатурну навігацію;
- focus-visible;
- aria-live для зворотного зв’язку;
- зрозумілий текст кнопок;
- достатній контраст;
- reduced motion;
- touch-first використання;
- відсутність горизонтального overflow;
- альтернативу для canvas/drag-only завдань.

---

## 11. PWA/offline checklist

Після зміни уроків, інструментів, JS або CSS:

- оновити `CACHE_NAME` у `sw.js`;
- додати нові HTML-сторінки уроків до `ASSETS`;
- додати нові JS/CSS/assets до `ASSETS`;
- перевірити, що app shell не падає offline;
- не кешувати випадкові тимчасові файли або профілі браузера.

Бажаний наступний крок: генерувати список `ASSETS` автоматично з каталогу уроків і файлової системи.

---

## 12. Тести перед здачею

Мінімум:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\generate-lesson-pages.ps1
powershell -ExecutionPolicy Bypass -File tests\run-tests.ps1
powershell -ExecutionPolicy Bypass -File tests\run-browser-smoke.ps1
```

Якщо PowerShell недоступний, агент має принаймні виконати еквівалентні статичні перевірки:

- усі JS-файли синтаксично валідні;
- немає `.browser-profile`;
- немає секретів;
- HTML-шаблон не дублює stylesheet/manifest;
- `sw.js` містить усі сторінки уроків;
- `catalog.js`, `landing-modules.js`, HTML-сторінки і генератор узгоджені.

## 13. Definition of Done

Зміна вважається завершеною лише якщо:

- немає службових профілів браузера та секретів;
- уроки відкриваються в student і teacher mode;
- немає довготривалого збереження дитячого прогресу;
- нові маршрути є в каталозі, генераторі та service worker;
- інтерактиви працюють без зовнішніх сервісів;
- зміна не створює горизонтальний overflow;
- документація оновлена;
- тести або їх еквівалентні перевірки пройдені;
- педагогічна складність відповідає 1–2 класу.

## 15. Заборонені типові помилки агентів

- Не переписувати весь проєкт без потреби.
- Не додавати фреймворк лише через звичку.
- Не додавати CDN “для швидкості”.
- Не зберігати відповіді дітей між сесіями.
- Не змішувати учнівський і вчительський текст.
- Не робити урок довшим, ніж потрібно.
- Не створювати новий тип активності, якщо є відповідне ядро.
- Не змінювати назви файлів без оновлення каталогу, генератора, HTML і service worker.
- Не додавати зовнішні iframe.
- Не залишати `.browser-profile` після локального браузерного тесту.
