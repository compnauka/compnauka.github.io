# Project Status

## Поточний стан

Проєкт `course/grade1` переведено з базового статичного макета у більш контрольований стан із початковим тестовим контуром, доступнісними виправленнями та захистом від проблем із кодуванням українських текстів.

### Що вже зроблено

- Додано автоматичні тести через `node --test`.
- Додано перевірки на mojibake/UTF-8-пошкодження українських текстів.
- Додано перевірки на базову HTML-структуру, skip link, PWA-обв’язку та відсутність inline `onclick`.
- Переписано сторінки `index.html`, `lesson_1_1.html`, `lesson_1_2.html`, `offline.html` у більш структурований вигляд.
- Додано `skip-link` і `id="main-content"` для кращої навігації.
- Для уроку `1.1` винесено JS у окремий файл `lesson_1_1.js`.
- Для головної сторінки додано окремий файл `main.js`.
- Додано `manifest.json`, `sw.js`, `offline.html`.
- Додано початковий словник перекладів `i18n.js`.
- Частину статичних і динамічних текстів переведено на `data-i18n` та `t(...)`.
- Для `canvas` додано доступніший сценарій використання і текстову альтернативу.
- Для drag-and-drop додано сценарій click/tap без обов’язкового перетягування.
- Додано `aria-live` для частини інтерактивного зворотного зв’язку.
- Додано базові дизайн-токени у `tokens.css`.
- Додано `focus-visible`, `prefers-reduced-motion`, `prefers-contrast`.

## Поточний тестовий контур

Тести зараз покривають:

- відсутність типових mojibake-маркерів;
- наявність критичних українських рядків у `i18n.js`;
- наявність `lang`, `title`, `skip-link`;
- правильний порядок підключення `i18n.js`;
- відсутність фальшивих `href="#"` і inline `onclick`;
- базові accessibility-маркери для уроку `1.1`;
- використання `i18n` у runtime-скриптах;
- базову PWA-обв’язку;
- наявність `focus-visible` і motion/contrast-адаптацій.

Поточний результат:

- `9/9` тестів проходять успішно.

## Відомі обмеження та залишкові ризики

### Архітектурні

- Проєкт усе ще залежить від `https://cdn.tailwindcss.com`.
- Іконки підтягуються з `cdnjs`.
- Шрифт підтягується через `@import` із Google Fonts.
- Токенізація стилів поки часткова, а не повна.
- Частина текстів у HTML ще не винесена в `i18n.js`.

### Доступність

- Доступність покращена, але ще не доведена до повного рівня WCAG 2.2 AA.
- Немає повної перевірки screen reader сценаріїв.
- Не додано окремий аудит контрастності всіх елементів.
- Не реалізовано повне покриття всіх сценаріїв клавіатурної взаємодії документованими тестами.

### PWA / Offline

- Є базовий service worker, але кешування ще дуже просте.
- Немає іконок для `manifest.json`.
- Немає окремої стратегії версіонування контенту уроків.
- Немає перевірок на офлайн-поведінку в браузері, лише структурна перевірка.

### Якість контенту

- `lesson_1_2.html` все ще заглушка.
- Частина головної сторінки все ще містить підготовчі картки без реального контенту.

## Що найкраще робити наступним кроком

Найкращий наступний крок:

## Крок 1. Прибрати найризиковішу зовнішню залежність з критичного шляху

Почати з відмови від `cdn.tailwindcss.com` у runtime.

### Чому саме це

- це зменшить залежність від мережі;
- покращить стабільність офлайн-режиму;
- прибере один із найбільших performance-ризиків;
- зменшить варіативність поведінки між середовищами;
- спростить контроль над стилями й відповідність внутрішнім стандартам.

### Рекомендований підхід

1. Зафіксувати фактично використані utility-класи.
2. Винести мінімально потрібні стилі у локальний CSS-шар.
3. Замінити CDN Tailwind на локальні стилі для `grade1`.
4. Прогнати тести після кожного малого кроку.

## Після цього

Другим кроком варто зробити повне винесення всіх видимих текстів у `i18n.js`.

### Це дасть

- ще кращий захист від поломок кодування;
- єдине джерело правди для текстів;
- простішу подальшу локалізацію;
- простіші тести на цілісність текстового контенту.

## Практичний порядок робіт

1. Прибрати `cdn.tailwindcss.com`, не ламаючи візуал.
2. Довинести тексти з HTML у `i18n.js`.
3. Додати іконки та завершити `manifest.json`.
4. Посилити service worker і кешування.
5. Розширити automated tests на локалізацію, прогрес уроку й офлайн-шар.
6. Провести окремий accessibility-прохід по клавіатурі, фокусу й screen reader сценаріях.
