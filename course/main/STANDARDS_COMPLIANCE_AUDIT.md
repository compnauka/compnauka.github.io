# Аудит відповідності сайту стандартам

Дата аудиту: 2026-04-05

Базовий документ для звірки: [PROJECT_STANDARDS.md](./PROJECT_STANDARDS.md)

## Висновок

Поточний стан сайту ближчий до робочого інтерактивного прототипу, ніж до версії, яка повністю відповідає власним стандартам проєкту.

Сильні сторони:

- є базова семантика `header` / `main`
- є `lang="uk"` на рівні HTML
- є локальне збереження прогресу
- у модульній версії вже частково враховано ARIA-атрибути для навігації та квізу

Основні проблеми:

- порушені базові вимоги WCAG 2.2
- відсутній PWA/offline-шар
- немає єдиної дизайн-системи на токенах
- у коді співіснують дві паралельні архітектури
- видимі тексти й стани ще не винесені в систему локалізації

Оцінка поточної відповідності:

- Семантика та базова структура: частково
- Доступність WCAG 2.2: частково / недостатньо
- Responsive та touch UX: частково
- Cognitive accessibility: частково
- Design tokens: не відповідає
- PWA та offline: не відповідає
- I18n та language governance: не відповідає
- Тестова готовність під стандарт: не відповідає

## Що знайдено в коді

### 1. Критичні невідповідності

#### 1.1 Заборонено масштабування сторінки

У файлі [index.html](./index.html) вказано:

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
```

Це прямо суперечить стандарту, де `user-scalable=no` заборонено.

Що треба зробити:

- прибрати `user-scalable=no`
- прибрати обмеження `maximum-scale=1.0`, якщо воно не має окремого обґрунтування
- залишити стандартний mobile viewport без блокування zoom

Пріоритет: `Critical`

---

#### 1.2 Відсутній PWA/offline-контур

У проєкті зараз немає:

- `manifest.json`
- `sw.js`
- `offline.html`
- реєстрації service worker
- офлайн-кешування shell і вже відкритих уроків

Це суперечить розділу `Performance & PWA Standards`.

Що треба зробити:

- додати `manifest.json` з `name`, `short_name`, `icons`, `start_url`, `display`, `lang`
- додати `sw.js` зі стратегіями `cache-first` для shell і `network-first` або `stale-while-revalidate` для контенту
- додати `offline.html`
- зареєструвати service worker при першому завантаженні
- забезпечити доступність вже відкритих уроків офлайн

Пріоритет: `Critical`

---

#### 1.3 Відсутня token-first дизайн-система

Зараз кольори, spacing, radius, тіні й частина типографіки задаються напряму в:

- [index.html](./index.html)
- [css/style.css](./css/style.css)
- [renderers.js](./renderers.js)

Також є inline-стилі на кшталт ширини прогресбарів.

Це суперечить вимозі мати єдине джерело істини у `tokens.css`.

Що треба зробити:

- створити `tokens.css`
- винести туди всі кольори, spacing scale, radius, shadows, font sizes, z-index
- замінити захардкожені значення на CSS custom properties
- прибрати inline visual styles там, де це можливо

Пріоритет: `Critical`

---

#### 1.4 Відсутній базовий accessibility-shell для всього застосунку

У коді не вистачає системних механізмів доступності:

- skip link до основного контенту
- глобального `aria-live` для повідомлень
- гарантованого `focus-visible`
- повноцінного `focus management`
- стилів для `prefers-reduced-motion`
- стилів для `prefers-contrast`

Що треба зробити:

- додати skip-link перед основним контентом
- зробити `<main>` фокусованим через `tabindex="-1"`
- додати глобальний елемент на кшталт `#announcements` з `aria-live="polite"`
- додати загальний стиль `:focus-visible`
- додати media query для `prefers-reduced-motion`
- додати media query для `prefers-contrast: high`

Пріоритет: `Critical`

## 2. Високопріоритетні розриви

### 2.1 Дві паралельні архітектури в одному проєкті

У репозиторії одночасно існують:

- велика inline-реалізація в [index.html](./index.html)
- окрема модульна реалізація в [js/main.js](./js/main.js), [renderers.js](./renderers.js), [contentLoader.js](./contentLoader.js), [utils.js](./utils.js)

Це створює ризики:

- дублювання логіки
- різний рівень доступності в різних гілках коду
- складність підтримки
- ризик виправляти не ту версію, яка реально є production-джерелом

Що треба зробити:

- вибрати одну канонічну архітектуру
- або повністю перейти на модульну версію
- або тимчасово лишити inline-версію, але прибрати дублікати
- зафіксувати це в README або в окремому ARCHITECTURE.md

Пріоритет: `High`

---

### 2.2 Немає унікального `document.title` для різних view

Зараз у [index.html](./index.html) стоїть статичний заголовок:

```html
<title>CS Mobile Course</title>
```

Але стандарт вимагає унікальний та описовий title для кожного view.

Що треба зробити:

- для landing: окремий title українською
- для home/curriculum: окремий title українською
- для lesson view: title з номером і назвою уроку
- оновлювати `document.title` при кожному переході

Пріоритет: `High`

---

### 2.3 Немає voice input для значущих текстових полів

У застосунку є щонайменше текстовий пошук, але немає voice input-патерну, який прямо вимагає стандарт.

Що треба зробити:

- додати кнопку голосового введення біля пошуку
- реалізувати `SpeechRecognition` / `webkitSpeechRecognition`
- коректно працювати з `aria-pressed`
- мати дружні повідомлення про стан прослуховування
- тихо приховувати кнопку, якщо API недоступне

Пріоритет: `High`

---

### 2.4 Порушення правил focus styling

У коді є `outline-none` на інтерактивних елементах, але немає єдиного гарантованого замінника на `:focus-visible`.

Що треба зробити:

- прибрати `outline-none`, якщо немає надійного аналога
- додати загальний фокус-індикатор із контрастом не менше 3:1
- окремо протестувати sticky-header сценарії, щоб focus не перекривався

Пріоритет: `High`

---

### 2.5 Видимі тексти не винесені в language resources

Тексти інтерфейсу зараз жорстко вшиті в шаблони та JS.

Це суперечить розділу `Internationalization & Language`.

Що треба зробити:

- винести всі UI-рядки в ресурсний файл, наприклад `locales/uk.json`
- створити простий словник `t(key)`
- винести назви кнопок, статусів, порожніх станів, помилок, підказок, навігації
- закласти запас під 30-40% розширення тексту

Пріоритет: `High`

## 3. Середньопріоритетні проблеми

### 3.1 Недостатня підтримка reduced motion

У проєкті є анімації, transition і декоративний motion, але немає повної гілки для `prefers-reduced-motion`.

Що треба зробити:

- додати universal override для animation/transition/scroll-behavior
- вимкнути декоративні анімації для користувачів із відповідною системною настройкою

Пріоритет: `Medium`

---

### 3.2 Недостатня підтримка high contrast

У стандарті є вимога враховувати `prefers-contrast: high`, але в коді цього немає.

Що треба зробити:

- додати окремі токени та стилі для high-contrast режиму
- перевірити кнопки, межі, focus ring, muted text, progress bars

Пріоритет: `Medium`

---

### 3.3 Невідповідність типографічним рекомендаціям

У [css/style.css](./css/style.css) використовується стек `Inter, system-ui...`, тоді як стандарт рекомендує dyslexia-friendly fonts, зокрема Atkinson Hyperlegible або Nunito.

Що треба зробити:

- перейти на один із рекомендованих шрифтів
- перевірити наявність повної кирилиці
- зберегти не більше двох typefaces на весь проєкт

Пріоритет: `Medium`

---

### 3.4 Технічні fallback-повідомлення не відповідають дитячому UX

У [contentLoader.js](./contentLoader.js) fallback-тексти технічні і більше підходять розробнику, ніж дитині або вчителю.

Що треба зробити:

- переписати error/loading/empty states дружньою українською
- пояснювати, що сталося, без технічного жаргону
- додати зрозумілу наступну дію, наприклад "Спробуй ще раз"

Пріоритет: `Medium`

## 4. Що вже зроблено добре

Нижче те, на що вже можна спиратись під час подальшого доведення до стандарту.

### 4.1 Базова семантика

У структурі сторінки вже є:

- `<header>`
- `<main>`
- кнопки замість клікабельних `div` у ключових місцях

Це хороша основа.

### 4.2 Є збереження прогресу

Прогрес зберігається в `localStorage`, що відповідає ідеї не втрачати результати дитини між сесіями.

### 4.3 У модульній версії є перші кроки до кращої a11y

У [renderers.js](./renderers.js) уже є:

- `aria-expanded`
- `aria-controls`
- `role="region"`
- `role="radiogroup"`
- `aria-live` для feedback у квізі

Це означає, що модульна архітектура потенційно краща база для доведення сайту до стандарту, ніж повністю inline-шаблон.

## 5. Практичний план робіт

### Етап 1. Must-have перед будь-яким глибшим UI-рефакторингом

- обрати одну канонічну архітектуру
- прибрати `user-scalable=no`
- зробити `main` фокусованим
- додати skip link
- додати глобальний `aria-live`
- додати `:focus-visible`
- додати `prefers-reduced-motion`
- додати динамічний `document.title`

Очікуваний результат:

- сайт перестає порушувати базові вимоги WCAG 2.2
- навігація стає більш прогнозованою
- базовий accessibility-shell готовий до наступних ітерацій

### Етап 2. Design system та стабілізація

- створити `tokens.css`
- винести кольори, spacing, radii, shadows, typography tokens
- прибрати hardcoded visual values
- уніфікувати компоненти кнопок, карток, прогресбарів, полів

Очікуваний результат:

- дизайн стає контрольованим і повторюваним
- нові зміни легше робити без регресій

### Етап 3. PWA та offline-first

- додати `manifest.json`
- додати `sw.js`
- додати `offline.html`
- кешувати shell і відкриті уроки
- показувати зрозумілий offline state

Очікуваний результат:

- сайт відповідає контексту шкільного використання
- уроки залишаються доступними за нестабільного інтернету

### Етап 4. I18n, дитячий UX і assistive features

- винести тексти в `locales/uk.json`
- переписати error/empty/loading states людською мовою
- додати voice input для пошуку та інших текстових полів
- перевірити tone of voice на відповідність дитячій аудиторії

Очікуваний результат:

- інтерфейс стає ближчим до цільової аудиторії
- знижується когнітивне навантаження

### Етап 5. Формальна перевірка проти стандарту

- keyboard-only walkthrough
- перевірка zoom 200%
- screen reader smoke test
- contrast audit
- Lighthouse
- ручний прогін PWA/offline сценаріїв

Очікуваний результат:

- проєкт можна оцінювати не "на око", а по факту проходження чекліста

## 6. Must-have / Should-have / Nice-to-have

### Must-have

- прибрати `user-scalable=no`
- уніфікувати архітектуру
- додати skip-link, `aria-live`, `focus-visible`, `tabindex="-1"` для `main`
- додати `prefers-reduced-motion`
- додати динамічні page titles
- створити `tokens.css`
- додати PWA/offline foundation

### Should-have

- додати voice input
- винести всі тексти в locale resources
- переписати всі стани помилок, завантаження, порожнього контенту
- додати `prefers-contrast`
- перейти на більш читабельний шрифт для дітей

### Nice-to-have

- окремий accessibility mode
- перемикач шрифту
- формалізований UX audit по 5-second test і heuristic evaluation
- автоматизований accessibility regression checklist

## 8. Короткий чекліст приймання

Після доопрацювань сайт можна вважати близьким до стандарту, якщо виконано таке:

- сторінка масштабується жестами та browser zoom
- усі основні сценарії проходяться тільки клавіатурою
- є видимий focus ring
- є skip-link
- є дружні `aria-live`-повідомлення
- рух можна зменшити через system preference
- інтерфейс працює офлайн хоча б для вже відкритих уроків
- всі візуальні значення йдуть через токени
- усі тексти інтерфейсу зберігаються в ресурсних файлах
- `document.title` змінюється відповідно до view

## 9. Підсумок

Щоб сайт максимально відповідав вашим стандартам, потрібно не точкове "підфарбування", а короткий, але системний етап стабілізації:

1. уніфікація архітектури
2. базова WCAG 2.2-обв'язка
3. token-first design system
4. PWA/offline foundation
5. i18n і дружні стани для дітей

Найкраща стратегія зараз: взяти модульну версію як основу, довести її до стандарту й поступово прибрати дублювання з inline-коду в `index.html`.
