# CHANGELOG_STANDARD.md

## Призначення

Цей документ задає єдиний стандарт ведення changelog для UI-змін у сервісах Офіс ПЛЮС.

Мета changelog:

- зафіксувати, що саме змінилося;
- зафіксувати, чому зміна була внесена;
- зафіксувати, на який стандарт або рев'ю спиралася зміна;
- спростити контроль якості, регресійне тестування і роботу агентів.

Changelog не є місцем для маркетингових формулювань. Це техніко-продуктовий документ.

---

## Де має лежати changelog

Рекомендована модель:

- глобальний `CHANGELOG_STANDARD.md` — у пакеті стандартів;
- у кожному сервісі — власний `CHANGELOG.md`.

Саме `CHANGELOG.md` сервісу ведеться за правилами цього документа.

---

## Коли changelog обов'язковий

Запис у changelog обов'язковий, якщо зміна зачіпає:

1. shell-рівень;
2. menu / toolbar / contextbar / statusbar;
3. keyboard shortcuts;
4. modal / dropdown / picker / popover;
5. responsive-поведінку;
6. workspace focus / selection / editing;
7. локальну специфікацію сервісу (`UI_STANDARD.md`);
8. міграційний план (`UI_MIGRATION_TO_STANDARD.md`).

Для дрібної внутрішньої технічної зміни без впливу на UI запис не обов'язковий.

---

## Формат версій

Рекомендований формат:

- `Unreleased`
- `1.0.0`
- `1.1.0`
- `1.1.1`

Якщо у сервісі поки немає стабільного versioning, допускається формат по датах:

- `2026-04-22`
- `2026-05-03`

Але для зрілих сервісів бажано перейти на semver.

## Значення секцій

### Added
Нова UI-можливість або новий стандартний елемент.

Приклади:
- додано contextbar для форматування фігур;
- додано режим inspector для молодшої школи;
- додано keyboard shortcut `Ctrl+D` для дублювання об'єкта.

### Changed
Зміна існуючої поведінки або структури.

Приклади:
- toolbar перебудовано за єдиним shell-стандартом;
- меню Малюнків змінено на `Файл · Редагування · Вставка · Інструменти · Перегляд · Допомога`;
- modal підтвердження перенесено до стандартного формату footer.

### Fixed
Виправлення помилки або невідповідності стандарту.

Приклади:
- виправлено відсутній focus-visible для canvas;
- виправлено закриття dropdown по Escape;
- виправлено неправильний порядок Undo/Redo.

### Removed
Свідомо прибраний UI-елемент або поведінка.

Приклади:
- прибрано дублювання zoom-controls у двох місцях;
- прибрано плаваючий тимчасовий toolbar;
- прибрано нестандартну локальну гарячу клавішу.

### Migration notes
Нотатки для розробника, агента або QA про те, що треба перевірити після зміни.

## Правила якості запису

Кожен запис повинен бути:

- конкретним;
- перевірюваним;
- коротким, але не розмитим;
- прив'язаним до реальної UI-зміни;
- придатним для QA.

### Погано
- покращено інтерфейс;
- виправлено деякі проблеми;
- оновлено дизайн.

### Добре
- toolbar ПЛЮС Тексту перебудовано: файлові дії та Undo/Redo тепер завжди стоять ліворуч;
- у workspace SVG додано `focus-visible` контур за стандартом `WORKSPACE_ACCESSIBILITY.md`;
- dropdown палітри кольорів тепер закривається по Escape і поза кліком за стандартом `DROPDOWN_STANDARD.md`.

## Обов'язкові посилання на джерело правди

Якщо зміна внесена для приведення до стандарту, у записі або в секції `Migration notes` бажано вказати, на що вона спирається:

- `ART_OFFICE_UI_STANDARD.md`
- `UI_STANDARD.md` сервісу
- `UI_MIGRATION_TO_STANDARD.md` сервісу
- `KEYBOARD_SHORTCUTS.md`
- `WORKSPACE_ACCESSIBILITY.md`
- `MODAL_STANDARD.md`
- `DROPDOWN_STANDARD.md`
- `UI_REVIEW_TEMPLATE.md` або конкретне рев'ю

## Обов'язкові типи змін, які треба фіксувати окремо

Не можна зливати в один розмитий пункт такі зміни:

1. shell-рівень;
2. keyboard shortcuts;
3. modal/dropdown behavior;
4. responsive-поведінка;
5. вікові адаптації;
6. критичні виправлення доступності.

Кожен такий блок має бути окремим пунктом.

```md

### Added
- Додано стабільно видимий inspector-блок у ПЛЮС Малюнки для режиму молодшої школи.

### Changed

### Removed
- Прибрано дубльований zoom-control у statusbar ПЛЮС Тексту.

### Migration notes
- Перевірити shell-відповідність через `COMPONENT_CHECKLIST.md`.

```

## Що не можна робити

Не можна:

- змішувати Added / Changed / Fixed в одному пункті;
- писати лише загальні фрази без перевірюваного змісту;
- пропускати зміни shell-рівня;
- не фіксувати accessibility-виправлення;
- записувати лише технічну деталь без UX-наслідку, якщо зміна видима користувачу.

---

## Правило для агента / розробника

Якщо агент або розробник вносить видимі UI-зміни, він повинен:

1. перевірити зміну за стандартами;
2. відобразити її в `CHANGELOG.md` сервісу;
3. сформулювати запис так, щоб QA міг перевірити його без читання diff;
4. у разі великої перебудови shell або interaction-документувати це окремими пунктами.

---

## Зв'язок із UI-рев'ю

Якщо зміна є наслідком офіційного UI-рев'ю, у `Migration notes` рекомендується вказувати:

- дату рев'ю;
- сервіс;
- ID проблеми з рев'ю.

Приклад:

```md
### Migration notes
- Виконано пункт `ATX-03` із `UI_REVIEW_art-text_2026-04-22.md`.
```
