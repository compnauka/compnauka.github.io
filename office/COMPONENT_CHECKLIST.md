# COMPONENT_CHECKLIST.md

# Чекліст перевірки UI-компонентів для всієї лінійки АРТ Офіс

Цей документ використовується під час:
- code review;
- ручного QA;
- роботи агента або розробника над будь-яким редактором лінійки;
- приймання нових функцій перед релізом.

Мета: гарантувати, що будь-які зміни не ламають єдиний UI-стандарт і не збільшують когнітивне навантаження.

---

## 1. Загальні правила приймання

Перед злиттям змін перевірити:

- [ ] інтерфейс відповідає `ART_OFFICE_UI_STANDARD.md`;
- [ ] сервіс відповідає власному `UI_STANDARD.md`;
- [ ] зміни не суперечать `UI_TOKENS.css` та `design-tokens.json`;
- [ ] збережено єдиний app shell: titlebar → menu bar → toolbar → contextual UI → workspace → statusbar;
- [ ] нова функція не додає дублювання команд у кількох місцях без потреби;
- [ ] усе часто вживане доступне максимум у 1–2 кліки;
- [ ] усе рідковживане винесене в меню або popover, а не на постійну панель.

---

## 2. Header / Titlebar

- [ ] хедер має висоту за стандартом;
- [ ] використано правильний сервісний акцентний колір;
- [ ] назва продукту читається одразу;
- [ ] назва документа/проєкту відображається стабільно;
- [ ] кнопки дій у хедері не дублюють toolbar без вагомої причини;
- [ ] хедер не стає вищим через локальні стилі;
- [ ] іконки та текст вертикально вирівняні;
- [ ] немає випадкових різних радіусів, тіней чи кнопок іншого стилю.

---

## 3. Головне меню

- [ ] порядок меню відповідає стандарту;
- [ ] базова рамка меню збережена;
- [ ] якщо сервіс не потребує специфічного пункту, меню не ускладнюється штучно;
- [ ] назви пунктів меню короткі та звичні;
- [ ] однакові команди мають однакові назви в усіх редакторах;
- [ ] гарячі клавіші показані однаково;
- [ ] меню не містить технічних або незрозумілих для користувача формулювань;
- [ ] меню не дублює повністю toolbar;
- [ ] меню зручно працює мишею і клавіатурою;
- [ ] menubar dropdown відповідає `DROPDOWN_STANDARD.md`.

---

## 4. Основна панель інструментів

- [ ] перші кнопки зліва всюди однакові: Новий, Відкрити, Зберегти, Скасувати, Повернути;
- [ ] групи інструментів відокремлені візуально;
- [ ] найчастіші дії залишені на основній панелі;
- [ ] панель не перетворена на перевантажений ribbon;
- [ ] однакові типи контролів мають однакову висоту;
- [ ] немає міксу дрібних і великих кнопок без системи;
- [ ] zoom / view-controls стоять праворуч або в стабільному місці за стандартом сервісу;
- [ ] toggle-кнопки мають чіткий active-state;
- [ ] hover, active, disabled узгоджені з design tokens;
- [ ] панель не розламується на випадкові другі рядки при стандартній ширині;
- [ ] якщо другий рядок дозволений сервісом, він описаний у стандарті.

---

## 5. Контекстна панель

- [ ] контекстна панель з'являється лише тоді, коли справді потрібна;
- [ ] у ній лише ті інструменти, що стосуються поточного виділення;
- [ ] контекстна панель не дублює постійно весь toolbar;
- [ ] виділення тексту, зображення, фігури, комірки чи блока дає різний, але передбачуваний набір дій;
- [ ] зникнення контекстної панелі не ламає layout;
- [ ] overlay-панелі не зсувають workspace, якщо це суперечить стандарту сервісу;
- [ ] для сервісів 1–5 класу contextual UI або sticky, або замінений на інший стабільний патерн.

---

## 6. Workspace / Робоча область

- [ ] робоча область займає максимум корисного простору;
- [ ] службові панелі не від'їдають забагато місця;
- [ ] полотно/документ/слайд/таблиця не зсунуті випадковими відступами;
- [ ] масштабування працює передбачувано;
- [ ] скроли лише там, де вони справді потрібні;
- [ ] overlay-елементи не перекривають критичний контент;
- [ ] для учня одразу зрозуміло, де основна зона роботи;
- [ ] workspace або його контейнер має focus-visible;
- [ ] фокус не губиться після закриття dropdown / popover / modal;
- [ ] поведінка відповідає `WORKSPACE_ACCESSIBILITY.md`.

---

## 7. Status Bar

- [ ] статус-бар має єдиний стиль лінійки;
- [ ] він не дублює меню та toolbar;
- [ ] він показує лише корисний стан: контекст, активний елемент, масштаб, режим;
- [ ] текст статус-бару короткий і читабельний;
- [ ] службова інформація не перевантажує нижню смугу;
- [ ] правий край статус-бару працює стабільно при зміні ширини вікна.

---

## 8. Кнопки, поля, списки, dropdown і popover

- [ ] кнопки мають стандартну висоту;
- [ ] всі dropdown/select мають однакову висоту і радіус;
- [ ] текст у кнопках не обрізається;
- [ ] іконки мають єдиний оптичний розмір;
- [ ] іконка без підпису використовується лише там, де значення очевидне;
- [ ] popover не зсуває layout, якщо має бути накладним;
- [ ] палітри, меню кольорів, shape-picker, stamp-picker мають єдиний стиль;
- [ ] закриття popover по кліку поза межами працює стабільно;
- [ ] клавіатурна навігація для списків і меню не зламана;
- [ ] dropdown відповідає `DROPDOWN_STANDARD.md`.

---

## 9. Модальні вікна

- [ ] модальне вікно має заголовок;
- [ ] структура modal є: header → body → footer;
- [ ] у footer не більше 3 кнопок;
- [ ] primary / secondary / danger дії розставлені послідовно;
- [ ] `Esc` і focus trap працюють коректно;
- [ ] backdrop click поводиться за правилом сценарію;
- [ ] modal відповідає `MODAL_STANDARD.md`.

---

## 10. Гарячі клавіші

- [ ] глобальні скорочення не перевизначені локально;
- [ ] `Ctrl+N`, `Ctrl+O`, `Ctrl+S`, `Ctrl+Z`, `Ctrl+Y`, `Ctrl+A` працюють за стандартом;
- [ ] сервісні скорочення відповідають `KEYBOARD_SHORTCUTS.md`;
- [ ] скорочення, показані в меню, реально працюють;
- [ ] `Esc` закриває верхній overlay-рівень послідовно.

---

## 11. Іконки та підписи

- [ ] в одній панелі не змішані несумісні іконографічні стилі;
- [ ] назви короткі;
- [ ] локалізація послідовна;
- [ ] підписи не дублюють зайве;
- [ ] немає технічного жаргону.

---

## 12. Підсумкове питання перед прийманням

- [ ] користувачу стало легше, а не складніше;
- [ ] сервіс став ближчим до єдиної лінійки;
- [ ] новий UI-патерн не створив виняток без документації;
- [ ] молодший учень не загубиться в контекстних панелях, якщо сервіс орієнтований на молодшу школу.
