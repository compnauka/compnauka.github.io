# UI_REVIEW_TEMPLATE.md

## Призначення

Цей документ задає єдиний шаблон для формалізованого UI-рев'ю будь-якого сервісу АРТ Офісу.
Шаблон використовується для:

- регулярного аудиту сервісу;
- приймання великих UI-змін;
- рев'ю роботи агента або розробника;
- фіксації відхилень від стандарту;
- планування наступної хвилі доопрацювань.

Документ не замінює `COMPONENT_CHECKLIST.md`, а працює разом із ним:

- `COMPONENT_CHECKLIST.md` — короткий операційний чекліст;
- `UI_REVIEW_TEMPLATE.md` — повний структурований звіт.

---

## Коли обов'язково проводити UI-рев'ю

UI-рев'ю обов'язкове, якщо:

1. змінено shell-рівень сервісу;
2. змінено меню, toolbar, contextbar або statusbar;
3. змінено keyboard interaction;
4. змінено modal / dropdown / picker / popover;
5. додано нову велику функцію;
6. змінено поведінку для мобільної або вузької ширини;
7. змінено логіку виділення, фокуса або редагування на полотні.

---

## Як називати файл рев'ю

Рекомендований формат:

`UI_REVIEW_<service-name>_<YYYY-MM-DD>.md`

Приклади:

- `UI_REVIEW_art-text_2026-04-22.md`
- `UI_REVIEW_art-paint_2026-05-03.md`

## Обов'язкова структура рев'ю

### 1. Заголовок

```md
# UI Review — [Назва сервісу]

Дата: YYYY-MM-DD
Версія сервісу: [version / commit / branch]
Автор рев'ю: [ім'я / агент / команда]
Тип рев'ю: повне / повторне / після релізу / перед релізом / тематичне
```

### 2. Контекст рев'ю

```md
## Контекст

- Що саме перевірялося
- Чому проводиться рев'ю
- Які документи були джерелом правди
```

Обов'язково перелічити нормативні документи, за якими оцінювався сервіс:

- `ART_OFFICE_UI_STANDARD.md`
- `UI_STANDARD.md` конкретного сервісу
- `UI_MIGRATION_TO_STANDARD.md` конкретного сервісу
- релевантні поведінкові стандарти (`KEYBOARD_SHORTCUTS.md`, `MODAL_STANDARD.md`, `DROPDOWN_STANDARD.md`, `WORKSPACE_ACCESSIBILITY.md`)

### 3. Короткий підсумок

```md
## Підсумок

Загальна оцінка: [відповідає / частково відповідає / не відповідає]
Рівень готовності: [production-ready / beta / MVP / needs rework]
Головний висновок: [1 абзац]
```

### 4. Сильні сторони

```md
## Що відповідає стандарту

- ...
- ...
- ...
```

Тут фіксуються лише ті рішення, які реально відповідають стандарту і варто зберегти.

### 5. Відхилення від стандарту

```md
## Відхилення від стандарту

### [ID] Коротка назва проблеми
- Рівень: blocker / high / medium / low
- Зона: shell / toolbar / menu / workspace / modal / dropdown / shortcuts / responsive / accessibility / content
- Джерело правди: [назва документа]
- Поточний стан: ...
- Як має бути: ...
- Ризик для користувача: ...
- Рекомендована дія: ...
```

Кожне відхилення повинно бути окремим блоком.

### 6. Вікові ризики

```md
## Вікові та педагогічні ризики

- Що може бути незрозумілим для молодших учнів
- Що створює зайве когнітивне навантаження
- Що допустимо для 6–11 класів, але не для 1–5 класів
```

Ця секція є обов'язковою для:

- АРТ Малюнки
- АРТ Схеми
- АРТ Слайди (якщо орієнтований на молодшу школу)

### 7. Доступність і keyboard-flow

```md
## Доступність та клавіатурна взаємодія

- focus-visible
- tab order
- escape behavior
- keyboard shortcuts
- screen-size behavior
```

### 8. Responsive і вузька ширина

```md
## Поведінка на вузьких екранах

- Що відбувається на 1280 px
- Що відбувається на 1024 px
- Що відбувається на 768 px
- Які елементи переносяться / ховаються / агрегуються
```

### 9. Пріоритетний план виправлень

```md
## Пріоритетний план виправлень

### P0 — Blockers
- ...

### P1 — High priority
- ...

### P2 — Medium priority
- ...

### P3 — Polish
- ...
```

### 10. Рішення по релізу

```md
## Рішення

- Можна випускати без змін / так / ні
- Можна випускати після P0 / так / ні
- Що саме має бути виправлено до злиття
```

## Обов'язкові критерії оцінки

Кожне повне рев'ю повинно оцінювати щонайменше ці 12 блоків:

1. Назва сервісу, titlebar і shell
2. Головне меню
3. Toolbar
4. Contextbar / inspector
5. Workspace
6. Statusbar
7. Keyboard shortcuts
8. Modal
9. Dropdown / menu / picker
10. Responsive
11. Accessibility
12. Відповідність локальному `UI_STANDARD.md`

## Шкала пріоритетів

### Blocker
Проблема, яка:
- ламає базовий сценарій;
- ламає відповідність shell-стандарту;
- створює серйозну непослідовність між сервісами;
- робить інтерфейс непридатним для цільової вікової групи.

### High
Проблема, яка не блокує реліз повністю, але істотно знижує якість, зрозумілість або цілісність лінійки.

### Medium
Проблема помірного впливу, яку треба виправити у найближчій хвилі.

### Low
Косметика або локальне полірування.

---

## Шкала відповідності стандарту

- **Відповідає** — критичних відхилень немає, допускаються лише low-пункти.
- **Частково відповідає** — shell- або поведінкові відхилення є, але сервіс можна довести без повного перероблення.
- **Не відповідає** — архітектура або поведінка конфліктує зі стандартом системно.

## Правило для агента / розробника

Агент або розробник не має права писати рев'ю у форматі довільного списку без цієї структури, якщо рев'ю позначене як офіційне.

Офіційне UI-рев'ю:
- має спиратися на документи стандарту;
- має фіксувати відхилення окремими блоками;
- має завершуватися пріоритетним планом виправлень;
- має містити рішення щодо релізу.
