# SHELL_COMPONENTS.md

## Призначення

Цей документ описує обов'язкові shell-компоненти для всієї лінійки АРТ Офіс. Його треба використовувати разом із:

- `ART_OFFICE_UI_STANDARD.md`
- `UI_TOKENS.css`
- `design-tokens.json`
- сервісним `UI_STANDARD.md`

Shell-компоненти — це не окремі віджети, а стабільний каркас програми. Користувач має відчувати знайому структуру незалежно від того, чи він працює в Тексті, Таблицях, Малюнках, Слайдах, Схемах або Векторі.

---

## 1. Обов'язковий каркас програми

Кожен сервіс має складатися з таких зон у такому порядку:

1. `titlebar`
2. `menu-bar`
3. `toolbar`
4. `contextual-ui` — опційний шар, який може бути mini-toolbar, context menu, inspector або sticky strip
5. `workspace`
6. `statusbar`

Заборонено:

- міняти порядок цих зон;
- змішувати головне меню і toolbar в один рядок;
- прибирати statusbar у desktop-версії;
- вставляти критично важливі дії прямо на полотно, якщо їм місце на toolbar або в стандартизованому contextual UI.

---

## 2. Titlebar

### Призначення

Показує користувачу:

- у якому редакторі він працює;
- з яким документом/проєктом він працює;
- де знаходяться глобальні дії верхнього рівня.

### Обов'язкові елементи

Ліворуч:
- бейдж або іконка сервісу;
- назва сервісу;
- назва документа / проєкту.

Праворуч:
- 1–3 глобальні дії верхнього рівня, якщо вони справді потрібні;
- зазвичай це: `Файл`, `Поділитися`, `Експорт`, `Профіль`.

### Правила

- titlebar не повинен дублювати toolbar;
- не виносити сюди часті команди форматування;
- не перевантажувати titlebar кнопками;
- використовувати акцентний колір сервісу як фоновий колір titlebar.

---

## 3. Menu bar

### Призначення

Містить повний набір функцій, включно з тими, що не винесені на toolbar.

### Базовий порядок меню

У всіх сервісах використовується така рамка:

`Файл · Редагування · Вставка · [специфічне меню] · Перегляд · Допомога`

### Дозволені специфічні меню

- `Формат`
- `Дані`
- `Слайд`
- `Інструменти`

### Правила

- меню повинно бути однаково розташоване у всіх сервісах;
- назви пунктів повинні бути максимально однакові між сервісами;
- команда з однаковим змістом повинна мати однакову назву всюди;
- не створювати екзотичні меню типу `Додатково`, `Ще`, `Налаштування полотна` без гострої потреби.

---

## 4. Toolbar

### Призначення

Toolbar містить лише найчастіші дії поточного сервісу. Він має бути швидким, коротким, передбачуваним.

### Базовий порядок груп

Зліва направо:

1. `Файлові дії`
2. `Історія`
3. `Основні інструменти або форматування`
4. `Специфічні інструменти сервісу`
5. `Вигляд / Zoom` — праворуч

### Обов'язкові правила

- `Новий`, `Відкрити`, `Зберегти` — завжди на початку toolbar;
- `Скасувати`, `Повернути` — одразу після файлових дій;
- `Zoom` і view-controls — у правій частині toolbar;
- групи мають відділятись вертикальним ритмом, а не різними стилями кнопок;
- toolbar не повинен перетворюватись на повний дубль menu bar.

### Чого не робити

- не змішувати кнопки режимів і кнопки форматування без групування;
- не дублювати рідкісні команди на toolbar, якщо вони є в menu bar;
- не вставляти в toolbar великі кастомні панелі, що ламають висоту shell.

---

## 5. Contextual UI

### Призначення

Дає користувачу інструменти, які мають сенс лише для поточного виділення або поточного локального сценарію.

### Дозволені реалізації

- `inline mini-toolbar`
- `context menu`
- `inspector`
- `sticky strip`

### Загальні правила

- contextual UI не повинен дублювати весь toolbar;
- за замовчуванням заборонено робити постійний другий ряд під toolbar, якщо він лише повторює toolbar;
- overlay-варіанти повинні відкриватися поруч із дією, а не займати місце shell без причини;
- для молодших учнів допустимі лише стабільні та передбачувані варіанти;
- для складних властивостей пріоритет має inspector.

### Приклади

- Текст → mini-toolbar для виділеного тексту;
- Таблиці → context menu або local popover для комірок і діапазонів;
- Малюнки → sticky strip або простий inspector;
- Слайди → mini-toolbar для тексту, inspector для фігур;
- Схеми → sticky strip або inspector;
- Вектор → inspector як основний патерн.

## 6. Workspace

### Призначення

Основна робоча область сервісу.

### Дозволені зони всередині workspace

- ліва сайдбар-зона;
- центральна робоча зона;
- права inspector-зона.

### Правила

- центральна зона завжди головна;
- плаваючі панелі не повинні зсувати документ/полотно без потреби;
- якщо сервіс має сайдбар, він повинен бути логічно необхідним;
- не використовувати більше двох постійних бокових панелей одночасно без вагомої причини.

### Допустимі патерни по сервісах

- `АРТ Текст`: центральна зона, без обов'язкового сайдбару;
- `АРТ Таблиці`: центральна зона + формульний рядок над таблицею;
- `АРТ Малюнки`: центральне полотно, опційно плаваюча палітра;
- `АРТ Слайди`: ліва панель мініатюр + центральний слайд;
- `АРТ Схеми`: ліва бібліотека блоків + центральне полотно;
- `АРТ Вектор`: центральне полотно, опційно ліва панель інструментів і права панель властивостей.

## 7. Statusbar

### Призначення

Дає короткий стан програми, але не дублює toolbar.

### Базовий шаблон

Ліворуч:
- контекст документа / об'єкта / виділення.

Праворуч:
- масштаб;
- режим вигляду;
- другорядні технічні стани.

### Приклади

- Текст: слова, символи, мова, масштаб;
- Таблиці: активна комірка, статистика виділення, масштаб;
- Малюнки: інструмент, координати, колір, масштаб;
- Слайди: номер слайда, вибраний об'єкт, масштаб;
- Схеми: вибраний блок, прив'язка, масштаб;
- Вектор: інструмент, координати, шар або вибір, масштаб.

### Правила

- statusbar має бути спокійним візуально;
- не робити з нього другий toolbar;
- не вставляти сюди критичні команди, що потрібні новачку для виживання.

---

## 8. Загальна поведінка shell

### Навігаційні правила

- користувач має знаходити однакові дії в однакових місцях;
- у всіх сервісах працює та сама логіка клавіатурних скорочень для базових дій;
- зміна сервісу не повинна ламати ментальну модель користувача.

### Візуальні правила

- однакова висота контролів;
- однаковий радіус кутів;
- однакові hover, active, focus, disabled;
- однакова щільність груп.

### Контентні правила

- прості, зрозумілі назви команд;
- українська локалізація всюди;
- без суміші стилів типу `Delete / Видалити / Remove` в різних місцях.

---

## 9. Мінімальний shell-чекліст для рев'ю

Перед релізом кожного сервісу перевірити:

- чи є titlebar, menu bar, toolbar, workspace, statusbar;
- чи toolbar починається з файлових дій та історії;
- чи масштаб знаходиться праворуч;
- чи однакові базові назви меню з іншими сервісами;
- чи критичні дії не заховані тільки в меню;
- чи contextual UI не дублює toolbar і не відбирає зайве місце у shell;
- чи сервіс виглядає як частина АРТ Офісу, а не як окрема випадкова сторінка.
