# WORKSPACE_ACCESSIBILITY.md

# Стандарт доступності та focus-visible для робочої області

Версія: 1.0  
Статус: обов'язковий нормативний документ

---

## 1. Призначення

Цей документ фіксує правила доступності для робочих областей АРТ Офіс:

- `contenteditable` у текстових документах;
- таблиця / grid;
- `canvas`;
- `svg`-workspace;
- комбіновані editor surface.

---

## 2. Базовий принцип

Робоча область — це не "фон", а керована інтерактивна зона.  
Користувач повинен у будь-який момент розуміти:

- чи має workspace фокус;
- де саме зараз фокус;
- як повернутися з toolbar / menu назад у робочу область;
- що станеться після `Esc`, `Enter`, `Tab`.

---

## 3. Focusable-вимоги

### 3.1. Що має бути focusable

Принаймні один з рівнів workspace повинен бути keyboard-focusable:

- контейнер документа;
- grid-контейнер;
- контейнер canvas;
- контейнер svg-полотна;
- сам активний `contenteditable`-елемент;
- активний block/object editor, якщо він є.

### 3.2. Мінімальна технічна вимога

Якщо native focus відсутній, контейнер має отримати `tabindex="0"`.

---

## 4. Focus-visible

### 4.1. Обов'язкове правило

Для workspace **обов'язковий** видимий `focus-visible`.

### 4.2. Вимоги до вигляду

- focus ring має бути помітним на будь-якому сервісному акценті;
- focus не можна прибирати через `outline: none` без повної заміни на видимий стан;
- фокус не повинен виглядати як помилка верстки.

### 4.3. Рекомендований стиль

- 2–3 px зовнішній ring;
- колір на основі акценту сервісу;
- радіус за формою workspace або активного контейнера.

---

## 5. Поведінка по сервісах

### 5.1. АРТ Текст
- активний документ або активний аркуш має мати видимий фокус;
- при завершенні роботи з toolbar фокус повинен повертатись у документ;
- якщо відкрите inline-редагування таблиці або зображення, фокус має бути в відповідному editable-контексті.

### 5.2. АРТ Таблиці
- активна комірка повинна мати видимий state незалежно від mouse / keyboard;
- grid-контейнер повинен мати зрозумілий keyboard state;
- рух стрілками не повинен губити фокус.

### 5.3. АРТ Малюнки
- полотно або контейнер полотна повинні мати focus-visible;
- після закриття палітри або picker фокус повертається до інструмента або до полотна;
- для молодших учнів стан "активне полотно" повинен бути очевидним.

### 5.4. АРТ Слайди
- активний слайд має бути фокусованим контейнером;
- при редагуванні тексту фокус переходить у текстовий блок;
- після `Esc` з текстового блоку фокус повертається на об'єкт або на слайд.

### 5.5. АРТ Схеми
- полотно має мати видимий стан фокуса;
- вибраний блок або конектор має окремий state виділення, але це не замінює focus-visible контейнера;
- після закриття dropdown/panel фокус повертається на canvas-container або обраний елемент.

### 5.6. АРТ Вектор
- svg/workspace container має бути focusable;
- виділений об'єкт має мати selection state;
- keyboard focus і selection state не можна зливати в один непомітний стан.

---

## 6. Повернення фокуса

### 6.1. Після overlay
Після закриття dropdown, picker, popover або невеликої контекстної панелі фокус повертається:
1. або на trigger;
2. або на workspace, якщо дія завершила взаємодію з trigger.

### 6.2. Після modal
Після закриття modal фокус повертається на елемент, який її відкрив.  
Якщо цей елемент більше не існує — на найближчий логічний контейнер workspace.

---

## 7. Клавіатурні сценарії

- `Tab` переводить фокус між shell-елементами;
- `Esc` закриває верхній overlay або завершує локальний режим;
- `Enter` підтверджує дію у відповідному контексті;
- стрілки повинні працювати там, де очікуються: у grid, menu, іноді на canvas/svg-сценаріях.

---

## 8. Текстові та нетекстові workspace

### 8.1. Текстові
Для `contenteditable`:
- caret повинен бути видимим;
- placeholder не повинен перекривати введення;
- selection state має бути стабільним.

### 8.2. Нетекстові
Для `canvas` / `svg`:
- має бути візуально зрозуміло, що поверхня активна;
- selection handles не замінюють focus ring;
- canvas не повинен бути "мертвим" з точки зору клавіатури.

---

## 9. Заборонено

- прибирати focus-visible без заміни;
- робити workspace недоступним із клавіатури, якщо сервіс підтримує keyboard interaction;
- після закриття overlay губити фокус у `body`;
- робити так, щоб користувач не розумів, де він знаходиться після `Esc`.
