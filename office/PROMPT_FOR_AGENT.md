# PROMPT_FOR_AGENT.md

# Системний промпт для агента або розробника, який працює над UI сервісів Офіс ПЛЮС

Скопіюйте цей текст у вашого агента, IDE-асистента або використовуйте як внутрішній регламент перед змінами в коді.

---

Ти працюєш над одним із сервісів лінійки **Офіс ПЛЮС** — браузерного набору шкільних редакторів, які мають виглядати як єдина екосистема.

Твоя задача — вносити зміни так, щоб:
- не ламати єдиний UI-стандарт;
- не збільшувати когнітивне навантаження;
- зберігати схожість між редакторами;
- робити інтерфейс більш професійним, але не важчим для дитини.

Перед будь-якими змінами ти зобов'язаний орієнтуватися на такі документи:

1. `ART_OFFICE_UI_STANDARD.md` — головний стандарт для всієї лінійки.
2. `UI_STANDARD.md` конкретного сервісу — локальний стандарт саме цього редактора.
3. `UI_MIGRATION_TO_STANDARD.md` конкретного сервісу — перелік того, що ще не відповідає стандарту.
4. `UI_TOKENS.css` — базові змінні, розміри та shell-поведінка.
5. `design-tokens.json` — машинозчитувані дизайн-токени.
6. `SHELL_COMPONENTS.md` — правила для каркаса інтерфейсу.
7. `KEYBOARD_SHORTCUTS.md` — єдиний стандарт гарячих клавіш.
8. `WORKSPACE_ACCESSIBILITY.md` — фокус, keyboard interaction і workspace accessibility.
9. `MODAL_STANDARD.md` — стандарт модальних вікон.
10. `DROPDOWN_STANDARD.md` — стандарт dropdown, menu і picker-поведінки.
11. `COMPONENT_CHECKLIST.md` — перевірка якості перед завершенням задачі.

## Головні принципи, яких ти маєш дотримуватися

### 1. Спочатку shell, потім функція
Будь-яка нова можливість має вбудовуватися у вже існуючий каркас:
- titlebar
- menu bar
- toolbar
- context bar / inspector
- workspace
- status bar

Ти не можеш довільно вставляти нові панелі, другі toolbar, сторонні блоки чи floating-елементи, якщо це не дозволено стандартом сервісу.

### 2. Однакові дії — в однакових місцях
Якщо дія є спільною для кількох редакторів, її місце, назва, стиль, keyboard shortcut та іконка мають бути максимально узгодженими.

Приклади:
- `Новий`, `Відкрити`, `Зберегти`, `Скасувати`, `Повернути` — зліва на основному toolbar.
- `Масштаб` — праворуч або в місці, закріпленому стандартом сервісу.
- `Перегляд`, `Допомога`, `Файл` — називаються однаково у всіх сервісах.

### 3. Не роби важкий ribbon
Офіс ПЛЮС — не копія Microsoft Office. Це шкільний пакет.

Тому:
- основна панель має містити лише часто вживане;
- рідковживане йде в меню або popover;
- інструменти форматування для вибраного об'єкта краще виносити в context bar;
- не створюй панелі з десятками кнопок, якщо їх можна згрупувати.

### 4. Часте ближче, рідке глибше
Перед додаванням нової кнопки спитай себе:
- це потрібно постійно чи інколи?
- це глобальна дія чи дія для вибраного елемента?
- це має бути в toolbar, context bar, menu чи popover?

### 5. Не дублюй без потреби
Одна й та сама функція не повинна бути одночасно:
- у header,
- у toolbar,
- у context bar,
- і ще окремо на canvas,
якщо це не критично для UX.

### 6. Overlay не повинен ламати layout
Палітри кольорів, меню фігур, shape-picker, stamp-picker, меню масштабу та інші допоміжні панелі мають по можливості накладатися поверх інтерфейсу, а не зміщувати workspace вниз або вбік.

### 7. Дитина має зрозуміти інтерфейс без інструктора
Мета — не максимальна функціональність будь-якою ціною, а керований, зрозумілий, чистий інтерфейс для школи.

### 8. Для молодших учнів контекст має бути стабільним
Якщо сервіс орієнтований на 1–5 клас, не ховай ключові параметри в contextual UI, що з'являється і зникає. У таких сценаріях використовуй sticky context panel або простий inspector.

## Обов'язковий порядок роботи над задачею

### Крок 1. Визнач сервіс
Спочатку зафіксуй, над яким сервісом іде робота:
- ПЛЮС Текст
- ПЛЮС Таблиці
- ПЛЮС Малюнки
- ПЛЮС Слайди
- ПЛЮС Схеми
- ПЛЮС Вектор

### Крок 2. Звір shell
Перед змінами перевір:
- чи є всі основні shell-зони;
- чи не зламано порядок меню;
- чи не порушено висоти, відступи, щільність та групування.

### Крок 3. Перевір поведінкові стандарти
Перед змінами перевір:
- чи не ламаються глобальні shortcuts;
- чи має workspace focus-visible;
- чи відповідають dropdown / picker поведінці стандарту;
- чи modal відповідає єдиній структурі.

### Крок 4. Визнач клас функції
Для будь-якої нової можливості визнач, до чого вона належить:
- глобальна дія;
- вставка;
- інструмент;
- форматування;
- дія над вибраним елементом;
- перегляд / режим;
- статус / інформація.

### Крок 5. Розмісти функцію правильно
Використовуй правило:
- глобальне → menu / toolbar;
- часте глобальне → toolbar;
- контекстне → context bar / inspector;
- рідковживане → menu / popover;
- службове → status bar.

### Крок 6. Перевір на дублювання
Після зміни перевір, чи не з'явилося зайве дублювання команд.

### Крок 7. Перевір за чеклістом
До завершення задачі проглянь `COMPONENT_CHECKLIST.md`.

### Крок 8. Онови документацію, якщо зміни системні
Якщо ти зробив зміну, що впливає на стандарт сервісу, онови:
- `UI_STANDARD.md`, якщо це нова норма;
- `UI_MIGRATION_TO_STANDARD.md`, якщо це крок міграції;
- `README.md`, якщо з'явився новий документ або змінилась структура пакета.

## Як приймати рішення щодо розміщення команд

### Меню
Використовуй меню для:
- повного переліку можливостей;
- імпорту/експорту;
- рідших команд;
- налаштувань перегляду;
- довідки.

### Toolbar
Використовуй toolbar для:
- найбільш частих дій;
- команд, що повинні бути “під рукою”;
- швидкого доступу до ключових сценаріїв.

### Context bar / Inspector
Використовуй для:
- форматування вибраного тексту;
- параметрів фігури, зображення, комірки, блока;
- дій, які не мають сенсу без виділення.

### Popover / Dropdown / Picker
Використовуй для:
- палітр кольорів;
- вибору форми;
- списків стилів;
- рідких параметрів;
- компактної подачі варіантів.

### Status bar
Використовуй лише для:
- стану документа;
- активного елемента;
- масштабу;
- координат;
- статистики виділення.

Не винось у status bar інтерактивні сценарії першого рівня.

## Спеціальні правила для всієї лінійки

### ПЛЮС Текст
- не перетворюй редактор на важкий “офіс для дорослих”;
- таблиці та зображення мають мати контекстне керування;
- текстове форматування не повинно розривати layout.

### ПЛЮС Таблиці
- формульний рядок є окремим елементом shell;
- групи “форматування” і “дані” мають бути візуально відділені;
- не змішуй статистику і формат числа в одну групу.

### ПЛЮС Малюнки
- основний фокус — швидке малювання без перевантаження;
- інструмент вибору є обов'язковим;
- палітри та pickers мають бути накладними, а не зсувати полотно;
- для молодших учнів параметри інструмента мають бути стабільно видимі або доступні в одному очікуваному місці.

### ПЛЮС Слайди
- постійна панель не повинна дублювати всі властивості об'єктів;
- форматування тексту, фігури й зображення бажано тримати контекстним;
- керування слайдами має бути окремим сценарієм.

### ПЛЮС Схеми
- ліва панель типів блоків є допустимим винятком;
- верхній toolbar має керувати схемою, а не дублювати вставку блоків;
- “Так/Ні”, маршрут, підпис та виділення мають бути добре видимі;
- для молодших учнів властивості вибраного блока краще тримати стабільно.

### ПЛЮС Вектор
- одна постійна панель краща за кілька перевантажених рядів;
- властивості об'єкта бажано переносити в context bar;
- прив'язка та масштаб мають бути стабільно доступні.

## Чого не можна робити

- не додавати нові елементи довільних розмірів;
- не використовувати інші радіуси, тіні, падінги без оновлення токенів;
- не змінювати порядок глобальних груп без сильної причини;
- не розміщувати технічні формати на головній панелі без користувацького контексту;
- не робити hover/active-states, що відрізняються від лінійки;
- не впроваджувати нові великі UI-патерни локально в одному сервісі без документації;
- не ускладнювати інтерфейс, якщо ту саму задачу можна розв'язати простіше;
- не ламати `Ctrl+S`, `Ctrl+Z`, `Esc`, `Tab` і workspace focus logic.

## Формат відповіді агента після виконання задачі

Після роботи агент повинен коротко повідомити:

1. Що саме змінено.
2. Які shell-зони зачеплено.
3. Чому обрано саме таке розміщення команд.
4. Чи є зміни в документах стандарту.
5. Які пункти з `COMPONENT_CHECKLIST.md` були критичними для цієї задачі.
6. Чи зачіпались `KEYBOARD_SHORTCUTS`, `WORKSPACE_ACCESSIBILITY`, `MODAL_STANDARD`, `DROPDOWN_STANDARD`.

## Критерій успіху

Завдання вважається виконаним добре, якщо після змін:
- сервіс виглядає ближчим до єдиної лінійки;
- інтерфейс не став важчим;
- користувачеві простіше знайти потрібні дії;
- нова функція не зламала shell;
- keyboard і focus-поведінка залишились послідовними;
- розробник, який зайде в код пізніше, побачить послідовну систему, а не набір винятків.
