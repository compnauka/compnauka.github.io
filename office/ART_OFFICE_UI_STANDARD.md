# Єдиний UI-стандарт лінійки АРТ Офіс

Версія: 1.3  
Статус: обов'язковий нормативний документ  
Сфера дії: АРТ Текст, АРТ Таблиці, АРТ Малюнки, АРТ Слайди, АРТ Схеми, АРТ Вектор

---

## 1. Призначення

Цей документ фіксує єдиний UI-стандарт для всієї лінійки АРТ Офіс.  
Будь-який розробник, дизайнер, агент або автоматизований інструмент, який змінює інтерфейс сервісів, **зобов'язаний** дотримуватись цього стандарту.

Мета стандарту:

- зменшити когнітивне навантаження при переході між сервісами;
- створити відчуття цілісного офісного пакета;
- зберегти простоту для учнів;
- наблизити продукти до професійного середовища без перевантаження;
- уніфікувати меню, панелі, статус-бари, розміри елементів і поведінку UI.

3. локальний стандарт сервісу.

## 3. Продуктові принципи

### 3.1. Обов'язкові принципи

1. **Однакова логіка в однакових місцях**  
   Команди `Новий`, `Відкрити`, `Зберегти`, `Скасувати`, `Повернути`, `Масштаб` розташовуються однаково в усіх сервісах.

2. **Постійна панель має бути короткою**  
   На постійній панелі лишаються лише часто вживані дії. Рідковживані дії не повинні займати цінне місце.

3. **Контекстні дії мають бути контекстними**  
   Форматування тексту, налаштування фігур, кольори, обтікання, шари, параметри об'єкта повинні з'являтись у контекстній панелі або в інспекторі лише коли це доречно.

4. **Жодних дублювань без причини**  
   Якщо команда вже є в лівій панелі, у контексті або в об'єктному меню, її не потрібно дублювати у верхньому toolbar, якщо дублювання не пришвидшує базовий сценарій.

5. **Професійність без ribbon-перевантаження**  
   Заборонено перетворювати інтерфейс на важкий багаторядковий ribbon із десятками одночасно видимих контролів.

6. **Дружність до школи**  
   Кнопки, написи, стани, кольори й підказки мають бути зрозумілими учням, учителям та адміністраторам.

7. **Поведінка важливіша за декоративні відмінності**  
   Однаковий досвід визначається не лише кольором і розмірами, а й гарячими клавішами, поведінкою меню, модалок, фокуса й workspace.

### 3.2. Вікове правило для контекстного UI

Для сервісів, основна аудиторія яких — учні 1–5 класів, контекстний UI:

- або **не використовується** як плаваючий нестабільний шар;
- або є **стабільно видимим** у вигляді `sticky strip`;
- або реалізується як простий inspector-блок чи коротка закріплена група властивостей.

Не допускається, щоб для молодших учнів основні параметри об'єкта ховались у панель, яка постійно з'являється і зникає без явного пояснення.

## 4. Єдина оболонка застосунку

Кожен сервіс повинен мати однаковий загальний каркас:

1. **Header / верхня шапка**
2. **Головне меню**
3. **Основна панель інструментів**
4. **Контекстний UI: mini-toolbar, context menu, inspector або sticky strip**
5. **Робоча область**
6. **Статус-бар**

### 4.1. Header

Header містить:

- логотип або піктограму сервісу;
- назву сервісу;
- назву документа / проєкту;
- індикатор стану збереження;
- за потреби — правий службовий блок.

Header не повинен містити велику кількість команд.  
Header — це область ідентичності та стану, а не робоча панель.

---

## 5. Стандарт головного меню

### 5.1. Базова рамка меню

Для більшості сервісів використовується такий каркас:

`Файл · Редагування · Вставка · [Інструменти] · [Предметне меню] · Перегляд · Допомога`

Де:

- `Файл` — завжди перший;
- `Редагування` — завжди другий;
- `Перегляд` — завжди передостанній;
- `Допомога` — завжди останній;
- `Вставка` — стандартне місце додавання нового об'єкта, фрагмента або елемента;
- `Інструменти` — використовується лише там, де сервіс має окремий режим вибору / малювання / редагування інструментом;
- `[Предметне меню]` — сервісний пункт, специфічний для редактора.

### 5.2. Каркас меню по сервісах

- **АРТ Текст** → `Файл · Редагування · Вставка · Формат · Перегляд · Допомога`
- **АРТ Таблиці** → `Файл · Редагування · Вставка · Формат · Дані · Перегляд · Допомога`
- **АРТ Малюнки** → `Файл · Редагування · Вставка · Інструменти · Перегляд · Допомога`
- **АРТ Слайди** → `Файл · Редагування · Вставка · Слайд · Перегляд · Допомога`
- **АРТ Схеми** → `Файл · Редагування · Вставка · Перегляд · Допомога`
- **АРТ Вектор** → `Файл · Редагування · Вставка · Формат · Перегляд · Допомога`

### 5.3. Вимоги до меню

- Висота меню: **34 px**
- Розмір шрифту: **13 px**
- Насиченість: **700**
- Горизонтальний відступ кнопки меню: **12 px**
- Кнопка меню повинна мати стани: `default / hover / open / focus-visible`
- Відкритий пункт меню повинен бути явно підсвічений
- Усі пункти меню мають мати однакову висоту рядка
- Меню не повинні "стрибати" при відкритті
- Усі dropdown-меню повинні відповідати `DROPDOWN_STANDARD.md`

### 5.4. Вміст меню за замовчуванням

#### Файл
Обов'язковий шаблон:

- Новий
- Відкрити…
- розділювач
- Зберегти
- Зберегти як…
- розділювач
- специфічні формати експорту / імпорту
- розділювач
- Друк…

#### Редагування
Обов'язковий шаблон:

- Скасувати
- Повернути
- розділювач
- Вирізати
- Копіювати
- Вставити
- розділювач
- специфічні редагувальні команди
- розділювач
- Виділити все

#### Перегляд
Обов'язковий шаблон:

- Масштаб −
- Масштаб 100%
- Масштаб +
- розділювач
- специфічні режими відображення

#### Допомога
Обов'язковий шаблон:

- Довідка / Як користуватись
- Клавіатурні скорочення
- Про сервіс

## 6. Стандарт основної панелі інструментів

### 6.1. Головне правило

Основна панель інструментів повинна читатись зліва направо як:

`Файл → Історія → Основні дії → Предметні швидкі команди → View / Zoom`

### 6.2. Обов'язкові групи

#### Лівий край — однаковий для всіх
1. `Новий`
2. `Відкрити`
3. `Зберегти`

#### Одразу після файлової групи
4. `Скасувати`
5. `Повернути`

#### Правий край — однаковий для всіх
- `−`
- `100%`
- `+`

За потреби біля масштабу допускаються:
- `Показ`
- `Прив'язка`
- `Сітка`
- `Підкладка`

### 6.3. Розміри панелі

- Висота основної панелі: **50 px**
- Висота кнопки-іконки: **34 px**
- Мінімальна ширина кнопки з текстом: **34 px**
- Радіус кнопки: **8 px**
- Відступ між кнопками всередині групи: **6 px**
- Відступ між групами: **10 px**
- Розділювач груп: висота **22 px**

### 6.4. Заборонені патерни

- панель у 3–4 ряди без гострої потреби;
- розміщення `Undo/Redo` праворуч, якщо в інших сервісах воно ліворуч;
- змішування файлових команд із кольорами та параметрами об'єкта;
- зберігання інструментів малювання в одному блоці з друком або масштабом.

## 7. Контекстний UI

### 7.1. Дозволені патерни

У лінійці АРТ Офіс дозволено лише такі патерни контекстного UI:

- `inline mini-toolbar`
- `context menu`
- `inspector`
- `sticky strip`

Детальна поведінка описана в `CONTEXTUAL_UI_STANDARD.md`.

### 7.2. Загальні правила

- контекстний UI не дублює весь toolbar;
- за замовчуванням заборонено робити постійний дублюючий другий ряд під toolbar;
- overlay-інструменти не повинні без потреби зсувати workspace;
- для молодших учнів контекстні властивості повинні бути стабільні, а не “миготливі”;
- складні властивості об'єкта потрібно виносити в inspector, а не в перевантажену mini-toolbar.

---

## 8. Workspace і доступність

Робоча область є повноправною інтерактивною зоною. До неї застосовуються вимоги з `WORKSPACE_ACCESSIBILITY.md`.

Короткі обов'язкові правила:

- workspace повинен мати передбачуваний фокус;
- `canvas`, `contenteditable`, `svg` або контейнер workspace мають бути keyboard-focusable, якщо взаємодія дозволяє роботу з клавіатури;
- для workspace обов'язковий видимий `focus-visible`;
- завершення редагування, скасування вибору або повернення до полотна мають бути послідовними між сервісами;
- втрата фокуса не повинна візуально робити робочу зону "зламаною".

---

## 9. Гарячі клавіші

Гарячі клавіші регулюються `KEYBOARD_SHORTCUTS.md`.

Короткі незмінні правила:

- `Ctrl+N` — новий
- `Ctrl+O` — відкрити
- `Ctrl+S` — зберегти
- `Ctrl+Z` — скасувати
- `Ctrl+Y` — повернути
- `Ctrl+A` — виділити все
- `Esc` — закрити меню / popover / діалог або скасувати поточний режим, якщо це дозволено сервісом

Локальний сервіс **не може** перевизначати ці скорочення без окремого документованого винятку.

---

## 10. Dropdown, popover і modal

### 10.1. Dropdown і menu
Усі dropdown-меню, менюбар-панелі, popover-picker та списки вибору повинні відповідати `DROPDOWN_STANDARD.md`.

### 10.2. Модальні вікна
Усі модальні вікна підтвердження, імпорту, експорту, налаштувань та попереджень повинні відповідати `MODAL_STANDARD.md`.

Без окремого дозволу не можна:

- створювати новий локальний стиль модалки;
- робити інший порядок кнопок;
- ламати `Escape` і keyboard focus trap;
- робити nested modal поверх modal.

---

## 11. Статус-бар

Статус-бар не дублює меню чи toolbar.  
Він відповідає лише на питання: **де я зараз і з чим працюю**.

### 11.1. Загальні правила

- висота: **26 px**
- шрифт: **12 px**
- ліворуч — контекст;
- праворуч — режим, масштаб, службові індикатори;
- текст короткий;
- статус-бар не повинен перетворюватись на окремий toolbar.

---

## 12. Розміри й токени

Базовими джерелами розмірів є:

- `UI_TOKENS.css`
- `design-tokens.json`

Ключові розміри:

- Header — `44 px`
- Menu bar — `34 px`
- Toolbar — `50 px`
- Sticky strip / fixed contextual strip — `38 px`
- Status bar — `26 px`
- Control height — `34 px`
- Radius default — `8 px`

Заборонено вводити нові локальні розміри shell-компонентів без оновлення токенів.

---

## 13. Критерій відповідності стандарту

Інтерфейс вважається таким, що відповідає стандарту, якщо:

1. shell відповідає єдиній структурі;
2. глобальні дії знаходяться в однакових місцях;
3. keyboard shortcuts узгоджені;
4. workspace має доступний focus-visible;
5. dropdown і modal поводяться за загальними правилами;
6. сервіс відповідає власному `UI_STANDARD.md`;
7. зміни проходять по `COMPONENT_CHECKLIST.md`.

---

## 14. Заборонено

- вигадувати новий shell для окремого сервісу;
- переносити `Undo/Redo` в інший бік без вагомої причини;
- ламати базовий порядок меню;
- робити приховану критичну функцію без видимої точки входу;
- робити dropdown, modal або workspace без keyboard-поведінки;
- ховати ключові параметри молодшої школи в нестабільний плаваючий context UI;
- вводити локальний дизайн токенів поза загальною системою.
